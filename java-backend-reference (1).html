04 Architecture

3-Layer Architecture

Kiến trúc 3 tầng là chuẩn của mọi Spring Boot project. Mỗi tầng chỉ làm 1 việc — tách rõ trách nhiệm.

1
Controller Layer
Nhận HTTP Request → Gọi Service → Trả Response. Không chứa logic.
@RestController
2
Service Layer
Toàn bộ business logic nằm đây: validate, tính toán, gọi repository.
@Service
3
Repository Layer
Tương tác database: CRUD, query, không biết gì về HTTP hay business logic.
@Repository
⚡ Request đi từ trên xuống: Client → Controller → Service → Repository → Database. Response đi ngược lại. Đây là luồng quan trọng nhất bạn phải nắm chắc.
05 Tầng 1

Controller Layer

Cửa ngõ vào ứng dụng. Nhận request, gọi service, trả response. Không được chứa logic nghiệp vụ.

UserController.java — CRUD đầy đủ

UserController.java 
@RestController                    ← = @Controller + @ResponseBody
@RequestMapping("/api/users")       ← base URL
@RequiredArgsConstructor           ← Lombok inject constructor
public class UserController {

    private final UserService userService; ← inject Service

    // GET /api/users → lấy tất cả users
    @GetMapping
    public ResponseEntity<List<UserResponse>> getAllUsers() {
        return ResponseEntity.ok(userService.findAll());
    }

    // GET /api/users/{id} → lấy 1 user
    @GetMapping("/{id}")
    public ResponseEntity<UserResponse> getUserById(
            @PathVariable Long id) {  ← lấy {id} từ URL
        return ResponseEntity.ok(userService.findById(id));
    }

    // POST /api/users → tạo user mới
    @PostMapping
    public ResponseEntity<UserResponse> createUser(
            @Valid @RequestBody UserRequest request) {
        //  @Valid = validate input, @RequestBody = parse JSON từ body
        UserResponse created = userService.create(request);
        return ResponseEntity.status(201).body(created);
    }

    // PUT /api/users/{id} → cập nhật user
    @PutMapping("/{id}")
    public ResponseEntity<UserResponse> updateUser(
            @PathVariable Long id,
            @Valid @RequestBody UserRequest request) {
        return ResponseEntity.ok(userService.update(id, request));
    }

    // DELETE /api/users/{id} → xóa user
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteUser(
            @PathVariable Long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build(); // 204
    }

    // GET /api/users?name=Minh → query param
    @GetMapping("/search")
    public ResponseEntity<List<UserResponse>> search(
            @RequestParam String name) { ← ?name=xxx
        return ResponseEntity.ok(userService.searchByName(name));
    }
}

06 Tầng 2

Service Layer

Não của ứng dụng. Toàn bộ logic nghiệp vụ nằm đây. Không biết HTTP tồn tại, chỉ nhận input và trả output.

UserService — Interface + Implement

UserService.java (interface) 
// Interface — định nghĩa contract
public interface UserService {
    List<UserResponse> findAll();
    UserResponse findById(Long id);
    UserResponse create(UserRequest request);
    UserResponse update(Long id, UserRequest request);
    void delete(Long id);
    List<UserResponse> searchByName(String name);
}
UserServiceImpl.java 
@Service                          ← đăng ký với Spring IoC
@RequiredArgsConstructor
@Transactional                    ← tự rollback nếu có lỗi
public class UserServiceImpl implements UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    @Override
    @Transactional(readOnly = true) ← chỉ đọc, tối ưu
    public List<UserResponse> findAll() {
        return userRepository.findAll()
            .stream()
            .map(this::toResponse)  ← Entity → DTO
            .toList();
    }

    @Override
    public UserResponse create(UserRequest request) {
        // 1. Validate business rule
        if (userRepository.existsByEmail(request.getEmail())) {
            throw new ConflictException("Email đã tồn tại");
        }

        // 2. Tạo Entity
        User user = User.builder()
            .name(request.getName())
            .email(request.getEmail())
            .password(passwordEncoder.encode(request.getPassword()))
            .role(Role.USER)
            .build();

        // 3. Lưu vào DB
        User saved = userRepository.save(user);

        // 4. Trả về DTO
        return toResponse(saved);
    }

    private UserResponse toResponse(User user) {
        return UserResponse.builder()
            .id(user.getId())
            .name(user.getName())
            .email(user.getEmail())
            .build();
    }
}

Repository Layer

Lớp duy nhất nói chuyện với database. Spring Data JPA tự generate SQL — bạn chỉ cần đặt tên method đúng cú pháp.

UserRepository.java

@Repository
public interface UserRepository
    extends JpaRepository<User, Long> {
    //                          ↑Entity  ↑ID type
    // JpaRepository cho bạn miễn phí:
    // save(), findById(), findAll(), delete(), count(), exists()...

    // TỰ SINH SQL bằng tên method (Method Query)
    Optional<User> findByEmail(String email);
    // → SELECT * FROM users WHERE email = ?

    List<User> findByNameContainingIgnoreCase(String name);
    // → SELECT * FROM users WHERE LOWER(name) LIKE %?%

    boolean existsByEmail(String email);
    // → SELECT COUNT(*) > 0 FROM users WHERE email = ?

    List<User> findByRoleOrderByCreatedAtDesc(Role role);
    // → SELECT * FROM users WHERE role = ? ORDER BY created_at DESC

    // JPQL — khi method query không đủ
    @Query("SELECT u FROM User u WHERE u.email = :email AND u.active = true")
    Optional<User> findActiveByEmail(@Param("email") String email);

    // Native SQL query khi cần hiệu năng tối đa
    @Query(value = "SELECT * FROM users WHERE created_at > NOW() - INTERVAL '7 days'",
           nativeQuery = true)
    List<User> findRecentUsers();
}

public class User {

    @LastModifiedDate
    private LocalDateTime updatedAt; ← tự cập nhật khi sửa

    // QUAN HỆ: 1 User có nhiều Post
    @OneToMany(mappedBy = "author", cascade = CascadeType.ALL)
    private List<Post> posts = new ArrayList<>();

JwtService.java

JwtService.java 
@Service
public class JwtService {

    @Value("${jwt.secret}")
    private String secretKey;

    @Value("${jwt.expiration}")
    private long expiration;

    // TẠO token từ user
    public String generateToken(User user) {
        return Jwts.builder()
            .subject(user.getEmail())
            .claim("role", user.getRole())
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(getSignKey())
            .compact();
    }

    // LẤY email từ token
    public String extractEmail(String token) {
        return parseClaims(token).getSubject();
    }

    // KIỂM TRA token có hợp lệ không
    public boolean isValid(String token, String email) {
        return extractEmail(token).equals(email)
            && !isExpired(token);
    }

    private boolean isExpired(String token) {
        return parseClaims(token).getExpiration().before(new Date());
    }
}

11 Security

Filter / Middleware

Filter chặn mọi request trước khi đến Controller. Dùng để verify JWT, log, rate limit.

JwtAuthFilter.java — Verify token mọi request

JwtAuthFilter.java 
@Component
@RequiredArgsConstructor
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws Exception {

        // 1. Lấy token từ header "Authorization: Bearer xxx"
        final String authHeader = request.getHeader("Authorization");

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response); ← không có token → đi tiếp
            return;
        }

        // 2. Tách token ra khỏi "Bearer xxx"
        final String token = authHeader.substring(7);
        final String email = jwtService.extractEmail(token);

        // 3. Verify và set vào SecurityContext
        if (email != null && SecurityContextHolder.getContext()
                .getAuthentication() == null) {

            UserDetails user = userDetailsService.loadUserByUsername(email);

            if (jwtService.isValid(token, email)) {
                // Token hợp lệ → cấp quyền truy cập
                UsernamePasswordAuthenticationToken authToken =
                    new UsernamePasswordAuthenticationToken(
                        user, null, user.getAuthorities());
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }

        // 4. Chuyển tiếp cho filter tiếp theo
        filterChain.doFilter(request, response);
    }
}

@ControllerAdvice — Bắt lỗi tập trung

GlobalExceptionHandler.java 
@RestControllerAdvice            ← bắt exception toàn app
public class GlobalExceptionHandler {

    // Bắt custom exception của mình
    @ExceptionHandler(AppException.class)
    public ResponseEntity<ErrorResponse> handleApp(
            AppException ex) {
        return ResponseEntity
            .status(ex.getStatusCode())
            .body(new ErrorResponse(ex.getMessage()));
    }

    // Bắt lỗi @Valid validation
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {

        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getFieldErrors()
            .forEach(e -> errors.put(e.getField(), e.getDefaultMessage()));

        return ResponseEntity.badRequest()
            .body(new ErrorResponse("Validation failed", errors));
    }

    // Bắt tất cả exception còn lại
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleAll(Exception ex) {
        return ResponseEntity.status(500)
            .body(new ErrorResponse("Internal server error"));
    }
}

@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PhoneValidator.class)
public @interface ValidPhone {

    Class<?>[] groups() default {};
    Class<?>[] payload() default {};
}

// Logic validate
public class PhoneValidator
    implements ConstraintValidator<ValidPhone, String> {

    @Override
    public boolean isValid(String phone, ConstraintValidatorContext ctx) {
        if (phone == null) return true; // null = ok (dùng @NotNull riêng)
        return phone.matches("^(0|\\+84)[0-9]{9}$");
    }
}

15 Tổng kết

Full Request Flow

Luồng đầy đủ của 1 request từ khi client gửi đến khi nhận response. Hiểu cái này = hiểu toàn bộ backend.

Bước 1 — Client
POST /api/posts với Authorization: Bearer {JWT} và JSON body trong HTTP request
Bước 2 — Filter Chain
JwtAuthFilter chặn request, đọc token từ header, verify với JwtService, nếu hợp lệ → set vào SecurityContext, đi tiếp
Bước 3 — Dispatcher Servlet
Spring MVC điều hướng request đến đúng method trong Controller dựa vào URL và HTTP method
Bước 4 — Controller
@PostMapping nhận request, @Valid validate DTO, nếu OK → gọi postService.create(request)
Bước 5 — Service
Chứa toàn bộ business logic: check quyền, validate nghiệp vụ, map DTO → Entity, gọi Repository để save
Bước 6 — Repository
JPA/Hibernate sinh SQL INSERT INTO posts..., thực thi với PostgreSQL, trả về Entity đã được gán ID
Bước 7 — Service trở lại
Map Entity → DTO Response (loại bỏ field nhạy cảm), trả về Controller
Bước 8 — Controller trở lại
ResponseEntity.status(201).body(response) → Spring tự convert sang JSON
Bước 9 — Client nhận
HTTP 201 Created với JSON body chứa post vừa tạo — nếu có lỗi ở bất kỳ bước nào, GlobalExceptionHandler bắt và format thành error response
Tầng 1
Phải tự hiểu
Luồng request, 3-layer, HTTP method/status, JWT flow — không AI nào thay thế được tư duy này
Tầng 2
Hiểu để dùng AI
Entity, Repository method query, Validation annotation — biết tên, biết dùng, AI gen code, mình review
Tầng 3
AI lo hoàn toàn
Boilerplate config, Docker, CI/CD, SecurityConfig phức tạp — chỉ cần biết mục đích, AI code cho

@Slf4j  ← Lombok tạo: private static final Logger log
public class UserServiceImpl implements UserService {

    public UserResponse create(UserRequest req) {

        // DEBUG — chi tiết khi dev (tắt trên production)
        log.debug("Creating user with email: {}", req.getEmail());

        if (userRepository.existsByEmail(req.getEmail())) {
            // WARN — cảnh báo, không crash nhưng đáng chú ý
            log.warn("Attempt to register with existing email: {}", req.getEmail());
            throw new ConflictException("Email đã tồn tại");
        }

        User saved = userRepository.save(/* ... */);

        // INFO — sự kiện quan trọng của business
        log.info("User created successfully. id={}, email={}",
            saved.getId(), saved.getEmail());

        return toResponse(saved);
    }

    public void delete(Long id) {
        try {
            userRepository.deleteById(id);
            log.info("User deleted. id={}", id);
        } catch (Exception e) {
            // ERROR — có lỗi thực sự, cần xem xét
            log.error("Failed to delete user. id={}, error={}", id, e.getMessage(), e);
            throw e;
        }
    }
}

Redis Template — thao tác thủ công

RedisTemplate trực tiếp 
@Service
@RequiredArgsConstructor
public class OtpService {

    private final RedisTemplate<String, String> redisTemplate;

    // Lưu OTP với TTL 5 phút
    public void saveOtp(String email, String otp) {
        String key = "otp:" + email;
        redisTemplate.opsForValue().set(
            key, otp,
            5, TimeUnit.MINUTES   ← tự xóa sau 5 phút
        );
    }

    // Xác thực OTP
    public boolean verifyOtp(String email, String otp) {
        String stored = redisTemplate.opsForValue().get("otp:" + email);
        if (otp.equals(stored)) {
            redisTemplate.delete("otp:" + email); ← dùng 1 lần rồi xóa
            return true;
        }
        return false;
    }

    // Rate limiting đơn giản
    public boolean isRateLimited(String ip) {
        String key = "ratelimit:" + ip;
        Long count = redisTemplate.opsForValue().increment(key);
        if (count == 1) // lần đầu: set TTL 1 phút
            redisTemplate.expire(key, 1, TimeUnit.MINUTES);
        return count > 100; ← quá 100 request/phút → block
    }
}

RabbitMQ — Message Queue

Queue = hàng đợi xử lý tác vụ nặng bất đồng bộ. Ví dụ: gửi email, resize ảnh, xuất PDF — không làm ngay mà đẩy vào hàng đợi để xử lý sau, không block request.

📬 Không dùng Queue: User đăng ký → đợi gửi email xong → trả response (chậm, có thể timeout). Dùng Queue: User đăng ký → đẩy task vào queue → trả response ngay → worker gửi email sau.

Config + Producer (gửi message)

RabbitMQ Config + Producer 
// pom.xml: spring-boot-starter-amqp

@Configuration
public class RabbitMQConfig {

    public static final String EMAIL_QUEUE    = "email.queue";
    public static final String EMAIL_EXCHANGE = "email.exchange";
    public static final String EMAIL_KEY      = "email.routing";

    @Bean
    public Queue emailQueue() {
        return new Queue(EMAIL_QUEUE, true); ← durable: không mất khi restart
    }

    @Bean
    public DirectExchange emailExchange() {
        return new DirectExchange(EMAIL_EXCHANGE);
    }

    @Bean
    public Binding emailBinding() {
        return BindingBuilder.bind(emailQueue())
            .to(emailExchange()).with(EMAIL_KEY);
    }
}

// PRODUCER — đẩy task vào queue
@Service
@RequiredArgsConstructor
public class EmailProducer {

    private final RabbitTemplate rabbitTemplate;

    public void sendWelcomeEmail(String email, String name) {
        EmailMessage msg = new EmailMessage(email, name, "WELCOME");
        rabbitTemplate.convertAndSend(
            RabbitMQConfig.EMAIL_EXCHANGE,
            RabbitMQConfig.EMAIL_KEY,
            msg
        );
        log.info("Email task queued for: {}", email);
        // Không đợi gửi email xong, trả về luôn!
    }
}

Consumer (xử lý message)

EmailConsumer.java 
@Service
@Slf4j
@RequiredArgsConstructor
public class EmailConsumer {

    private final JavaMailSender mailSender;

    // Lắng nghe queue, tự xử lý khi có message
    @RabbitListener(queues = RabbitMQConfig.EMAIL_QUEUE)
    public void handleEmailTask(EmailMessage message) {
        log.info("Processing email for: {}", message.getEmail());

        try {
            SimpleMailMessage mail = new SimpleMailMessage();
            mail.setTo(message.getEmail());
            mail.setSubject("Chào mừng "    + message.getName());
            mail.setText("Cảm ơn bạn đã đăng ký!");
            mailSender.send(mail);

            log.info("Email sent successfully to: {}", message.getEmail());

        } catch (Exception e) {
            log.error("Failed to send email to: {}", message.getEmail(), e);
            // Message sẽ vào Dead Letter Queue (DLQ)
            // để retry hoặc xem xét sau
            throw e;
        }
    }
}

// Data class cho message
@Data @AllArgsConstructor @NoArgsConstructor
public class EmailMessage implements Serializable {
    private String email;
    private String name;
    private String type; // WELCOME, RESET_PASSWORD, OTP...
}

FROM eclipse-temurin:17-jre-alpine

GitHub Actions — .github/workflows/deploy.yml

.github/workflows/deploy.yml 
name: Build and Deploy

on:
  push:
    branches: [ main ]          # chạy khi push vào main

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      # 1. Checkout code
      - uses: actions/checkout@v4

      # 2. Cài Java 17
      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      # 3. Cache Maven dependencies
      - name: Cache Maven
        uses: actions/cache@v4
        with:
          path: ~/.m2
          key: maven-${{ hashFiles('**/pom.xml') }}

      # 4. Build và Test
      - name: Build with Maven
        run: mvn clean package -DskipTests=false
        # Test phải pass mới deploy!

      # 5. Build Docker image
      - name: Build Docker image
        run: |
          docker build -t myapp:${{ github.sha }} .
          docker tag myapp:${{ github.sha }} myuser/myapp:latest

      # 6. Push lên Docker Hub
      - name: Push to Docker Hub
        run: |
          echo "${{ secrets.DOCKERHUB_TOKEN }}" | docker login -u myuser --password-stdin
          docker push myuser/myapp:latest

      # 7. SSH vào server và deploy
      - name: Deploy to server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            docker pull myuser/myapp:latest
            docker-compose -f /app/docker-compose.prod.yml up -d --no-deps app
            docker system prune -f
🔐 Tất cả secret (DOCKERHUB_TOKEN, SERVER_HOST, SERVER_SSH_KEY) lưu trong GitHub → Settings → Secrets. KHÔNG bao giờ hardcode trong file.

Microservices — Tách nhỏ service

Monolith = 1 app to làm tất cả. Microservices = nhiều app nhỏ, mỗi app làm 1 việc. Phù hợp khi hệ thống lớn, nhiều team. Bắt đầu luôn bằng monolith, scale lên micro khi cần.

Kiến trúc Microservices

Cấu trúc services 
// Thay vì 1 monolith to:
my-app/
  ├── controller/
  ├── service/
  └── repository/

// Tách thành nhiều service độc lập:
api-gateway/        ← cổng vào duy nhất (port 80)
  ↓ điều hướng đến:
user-service/       ← quản lý user, auth (port 8081)
product-service/    ← quản lý sản phẩm (port 8082)
order-service/      ← quản lý đơn hàng (port 8083)
payment-service/    ← xử lý thanh toán (port 8084)
notification-svc/   ← gửi email/SMS (port 8085)

// Mỗi service có database riêng!
// Giao tiếp qua REST hoặc Message Queue

API Gateway + Service Discovery

Spring Cloud Gateway 
// API Gateway — điều hướng request
// pom.xml: spring-cloud-starter-gateway

spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service   ← load balanced
          predicates:
            - Path=/api/users/**  ← chuyển đến user-service

        - id: product-service
          uri: lb://product-service
          predicates:
            - Path=/api/products/**

// Gọi service khác từ Order Service
@Service
public class OrderService {

    // WebClient — gọi HTTP đến service khác
    private final WebClient webClient;

    public ProductDto getProduct(Long productId) {
        return webClient
            .get()
            .uri("http://product-service/api/products/" + productId)
            .retrieve()
            .bodyToMono(ProductDto.class)
            .block();
    }

    public void placeOrder(OrderRequest req) {
        // Sau khi đặt hàng → publish event cho notification-service
        rabbitTemplate.convertAndSend("order.placed", req);
    }
}

Monolith vs Microservices — Khi nào dùng cái nào?

✅ Monolith — Dùng khi:
• Team nhỏ (dưới 10 người)
• Product mới, chưa rõ scale
• Startup, MVP, side project
• Deadline gấp
• Không có DevOps chuyên nghiệp
✅ Microservices — Dùng khi:
• Team lớn, nhiều team độc lập
• Scale từng phần khác nhau
• Yêu cầu availability cao
• Đã có monolith và đang scale vấn đề
• Có đội DevOps/Platform Engineering
💡 Lời khuyên thực tế: Bắt đầu bằng Monolith được thiết kế tốt (module hóa rõ ràng). Khi thực sự cần scale → tách từng module ra thành service riêng. Đừng bắt đầu bằng Microservices khi còn là junior.
